## 1. 背景与目标

### 1.1 现有问题

- 原设计仅针对"研发流水线"场景，Pipeline 固定写死
- 复用性差：无法适配营销团队、短剧团队等不同业务场景
- 技术骨架尚未实现，代码从零开始

### 1.2 核心洞察

> **Claude Code = 通讯平台**（人在指挥 AI 干活，流程随机）
>
> **Agent Teams Harness = AI 福特工厂**（流水线装配，多角色协作，结果导向）

**关键区别**：
- 不强调流式输出，强调**结果产出和传递**
- 团队在现实世界有**真人映射**，不是纯 AI 对话
- 流程是**可配置的**，像搭乐高一样灵活

### 1.3 设计目标

**定位升级**：AI 流水线工厂（而非 AI 编程助手）

| 目标 | 说明 |
|------|------|
| **可配置 Pipeline** | Pipeline = 一串 Stage，可视化编辑 + JSON Schema 版本控制 |
| **Artifact 驱动** | 流水线中传递的是"产出物"，而非对话消息 |
| **场景通用** | 研发/营销/短剧均可使用同一套框架，只需配置不同 Pipeline |
| **AI + 人工混合** | 每个 Stage 可选：纯 AI / 纯人工 / AI+人工协作 |

## 2. 核心抽象

### 2.1 核心概念

```
Pipeline (可配置流水线)
├── Stage 1: [输入] → [处理模式] → [输出: Artifact]
├── Stage 2: [输入: Artifact] → [处理模式] → [输出: Artifact]
└── ...
```

| 概念 | 说明 | 示例 |
|------|------|------|
| **Artifact** | 流水线中传递的产出物 | 情报 Item、意向 Prospect、特性 Feature、需求 Requirement |
| **Stage** | 装配工位，定义输入/处理方式/输出 | 情报收集 Stage、意向分析 Stage |
| **StageHandler** | Stage 的执行逻辑 | AI Handler（自动）、Human Handler（人工）、Hybrid Handler（AI+人工） |
| **Pipeline** | 一串 Stage 串起来的完整流水线 | 研发 Pipeline、营销 Pipeline |
| **Agent** | 附着在 Stage 上的执行者（可为 AI 或真人） | 情报 AI、售前分析师、产品经理 |

### 2.2 处理模式

| 模式 | 说明 | 适用场景 |
|------|------|----------|
| **AI_AUTO** | AI 自动处理，无需人工介入 | 情报提取、格式转换、数据清洗 |
| **HUMAN_ONLY** | 纯人工处理，AI 仅做辅助 | 需求评审、架构审批、UAT 验收 |
| **AI_DRAFT_HUMAN_APPROVE** | AI 出草稿，人工确认/修改 | 意向分析、特性评审、方案选择 |

### 2.3 Artifact 类型定义

```json
{
  "artifact_types": [
    { "id": "raw_intel", "name": "原始情报", "stages": ["情报收集"] },
    { "id": "intel_item", "name": "情报条目", "stages": ["情报分析"] },
    { "id": "prospect", "name": "意向", "stages": ["意向分析"] },
    { "id": "feature", "name": "特性", "stages": ["特性规划"] },
    { "id": "requirement", "name": "需求", "stages": ["需求拆分"] },
    { "id": "spec", "name": "技术规格", "stages": ["架构设计"] },
    { "id": "code", "name": "代码", "stages": ["编码"] },
    { "id": "test_report", "name": "测试报告", "stages": ["测试"] }
  ]
}
```

## 3. Pipeline 配置设计

### 3.1 配置方案

- **底层**：JSON Schema 定义 Pipeline 结构，可 Git 版本控制
- **上层**：Web UI 可视化编辑器（拖拽配置 Pipeline）
- **飞书**：作为数据源/通知渠道集成，不做配置载体

### 3.2 Pipeline Schema

```json
  "pipeline": {
    "id": "rd-pipeline",
    "name": "研发流水线",
    "description": "从情报到上线的完整研发流程",
    "version": "1.0.0"
  },
  "stages": [
    {
      "id": "stage-intel-collection",
      "name": "情报收集",
      "order": 1,

    {
      "id": "stage-intel-analysis",

      "id": "stage-feature-planning",
      "name": "特性规划",
      "order": 3,
      "input": { "type": "prospect" },

```

### 3.3 场景示例

**研发 Pipeline**：
```
情报收集(AI) → 情报分析(AI+人工) → 意向确认(人工) → 特性规划(人工) → 需求拆分(AI+人工) → 架构设计(AI+人工) → 编码(AI) → 测试(AI) → UAT(人工) → 发版
```

**营销 Pipeline**：
```
热点挖掘(AI) → 议题审核(人工) → 内容制作(AI) → 审核校对(人工) → 分发执行(AI+人工)
```

**短剧 Pipeline**：
```
创意征集(AI) → 剧本评审(人工) → 分镜生成(AI) → 拍摄统筹(人工) → 后期制作(AI) → 内容审核(人工) → 发布
```

## 4. 技术架构

### 4.1 分层架构

```
┌─────────────────────────────────────────────────────────┐
│                    Web UI 层                             │
│  Pipeline 编辑器 │ 控制台 │ 产出物查看器                  │
└─────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────┐
│                  Pipeline 配置层                         │
│  Pipeline Schema 验证 │ 配置解析 │ 版本管理               │
└─────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────┐
│                  Pipeline 执行引擎                        │
│  Stage 编排器 │ Artifact 传递 │ 状态管理                   │
└─────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────┐
│                    Stage 执行层                          │
│  AI Handler │ Human Handler │ Hybrid Handler            │
└─────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────┐
│                    工具层                                │
│  Tool Registry │ 内置工具 │ MCP 扩展                     │
└─────────────────────────────────────────────────────────┘
```

### 4.2 核心组件

| 组件 | 职责 |
|------|------|
| **PipelineConfig** | Pipeline 配置的内存表示，包含 stages 列表 |
| **PipelineExecutor** | Pipeline 编排器，负责按顺序执行各 Stage |
| **StageExecutor** | Stage 执行器，根据 handler 类型调用不同处理逻辑 |
| **ArtifactStore** | 产出物持久化，支持版本和变更历史 |
| **ToolRegistry** | 工具注册中心，AI Agent 可用的工具集 |
| **AgentLoop** | 通用 AI 代理循环（借鉴 Claude Code 设计） |

### 4.3 Agent Loop 设计（借鉴 Claude Code）

```
用户输入 / 系统触发
        ↓
   ┌─────────────────────────────────────┐
   │         消 息 列 表 [messages]        │
   └─────────────────────────────────────┘
        ↓
   ┌─────────────────────────────────────┐
   │           LLM API 调用                │
   └─────────────────────────────────────┘
        ↓
   ┌─────────────────────────────────────┐
   │     LLM 响应 (stop_reason)            │
   └─────────────────────────────────────┘
        ↓
   ┌──────────────┬──────────────────────┐
   │ tool_use?    │ text/.stop?
   └──────────────┴──────────────────────┘
        ↓                    ↓
   执行工具（Registry）    返回结果
   追加 tool_result
        ↓
    循环回退 → messages[]
```

**与 Claude Code 的区别**：
- 不强调流式输出（text streaming）
- 强调**结果持久化**（Artifact Store）
- 强调**人工介入点**（Human Handler）

## 5. 落地计划

### Phase 1: 跑通情报→意向闭环（当前）

**目标**：验证核心架构可行性

**交付物**：
1. Pipeline Schema 定义（JSON Schema）
2. PipelineExecutor 框架代码
3. Stage Handler 基础接口（AI / Human / Hybrid）
4. ToolRegistry 基础实现
5. 情报收集 Stage 示例（接入 china-demand-mining skill）
6. 意向分析 Stage 示例（AI+人工协作）
7. 简化版 Web UI Pipeline 编辑器

**验收标准**：
- 能通过 JSON 配置加载 Pipeline
- 情报收集 Stage 能自动从多渠道抓取情报
- 意向分析 Stage 能 AI 出草稿 + 人工确认
- 产出物能在 Stage 之间正确传递

### Phase 2: 完善特性→需求链路

**目标**：跑通 FR-010 需求拆分工作流

### Phase 3: 通用化 + 模板市场

**目标**：支持多场景 Pipeline

## 6. 与原设计的差异

| 项目 | 原设计 (v1.x) | 新设计 (v2.0) |
|------|--------------|--------------|
| 定位 | AI 研发助手 | AI 流水线工厂 |
| Pipeline | 固定 7 个 Agent | 可配置 Stage 串 |
| 场景 | 仅研发 | 研发/营销/短剧通用 |
| 配置 | 文档描述 | JSON Schema + Web UI |
| 执行 | 未实现 | PipelineExecutor 框架 |
| 产出物 | 意向/特性/需求 | Artifact（统一抽象） |
