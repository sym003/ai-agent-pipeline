## 1. 产品概述

### 1.1 产品定位

Agent Teams Harness 是一个**企业级 AI Agent 协作流水线平台**，将传统软件研发中从商机捕获到产品上线的完整流程，通过多个专业 AI Agent 团队实现自动化协作，与飞书多维表格深度集成，实现研发流程的智能化升级。

### 1.2 目标用户

| 角色 | 描述 | 使用场景 |
|------|------|---------|
| **产品负责人** | 管理产品规划和优先级 | 审批意向、确认特性优先级 |
| **产品设计师** | 负责需求细化和验收 | 需求评审、设计验收 |
| **研发工程师** | 负责架构设计和编码 | 架构评审、代码协作 |
| **测试工程师** | 负责测试用例和验收 | 测试结果确认 |
| **售前人员** | 负责商机和客户 | 录入商机、跟进意向 |

### 1.3 产品愿景

**让每个创意从灵感到上线，都有一支 AI 团队护航。**

### 1.4 核心价值

1. **效率提升**：AI 自动完成信息收集、分析评估、文档编写等重复性工作
2. **质量保障**：标准化流程 + SDD 文档规范 + AI 审查
3. **全程可追溯**：飞书多维表格作为唯一真实来源，所有状态流转有据可查
4. **人机协作**：AI 做好基础工作，关键决策由真人把关

### 3.9 Agent 入驻设计（Agent Onboarding）

#### FR-015 Agent 入驻流程

| ID | 功能点 | 优先级 | 说明 |
|----|--------|--------|------|
| FR-016-1 | 注册接口 | P0 | Agent 调用 POST /api/agents/register → 获得 api_key + claim_token，状态变为 pending_claim |
| FR-016-2 | 验证码发送 | P0 | 系统向 Agent 的邮箱发送 6 位验证码（10分钟有效） |
| FR-016-3 | 验证码认领 | P0 | Agent 调用 POST /api/agents/verify 输入验证码 → 状态变为 active |
| FR-015-4 | 重新发送 | P1 | Agent 可调用 API 重新发送验证码（覆盖旧码） |

#### FR-016 Agent 状态与角色

| ID | 功能点 | 优先级 | 说明 |
|----|--------|--------|------|
| FR-016-1 | 状态枚举 | P0 | Agent 状态：pending_register（未注册）/ pending_claim（待认领）/ active（活跃）/ suspended（暂停） |
| FR-016-2 | 角色枚举 | P0 | Agent 角色：intelligence / presales / product_owner / requirement / architect / coder / tester / fallback |

## 5. 用户场景

### 5.1 场景一：商机自动录入

```
角色: 售前人员
触发: 在飞书商机跟进表中新增一条商机记录

流程:
1. 售前人员在飞书商机跟进表中填写商机信息
   - 客户名称：XX 科技
   - 需求描述：需要一套智能客服系统
   - 联系方式：xxx@xx.com
   
2. 飞书自动化触发器检测到新增记录
3. 系统自动创建 Intent 对象，状态设为"新意向"
4. 发送邮件通知售前分析 AI 团队
5. 售前人员收到邮件确认：已自动录入意向池

预期结果: 商机从飞书表格自动流入 Agent Teams 流水线
```

### 5.2 场景二：售前自动评估

```
角色: 售前分析 AI 团队
触发: 新意向进入评估状态

流程:
1. AI 团队读取商机信息
2. 从三个维度评估：
   - 技术可行性：我们是否有相关技术积累？（查询现有产品能力）
   - 商业价值：市场规模多大？（查询市场数据）
   - 客户需求强度：客户需求是否明确？（分析需求描述）
   
3. AI 输出评估报告：
   - 技术可行性: ★★★★☆ (4/5)
   - 商业价值: ★★★★★ (5/5)
   - 客户需求强度: ★★★★☆ (4/5)
   - 建议优先级: P0（高价值）
   
4. 自动更新飞书表格的优先级字段
5. 发送邮件通知产品负责人

预期结果: 自动完成评估，更新优先级，通知决策者
```

### 5.3 场景三：产品负责人审批

```
角色: 产品负责人
触发: 收到待审批邮件

流程:
1. 产品负责人收到邮件：「【待审批】XX科技-智能客服系统 - P0-高价值」
2. 点击邮件中的链接打开飞书表格
3. 查看售前分析评估结果
4. 结合产品路标和 ROI 分析
5. 在飞书表格中将状态更新为"已批准进入特性池"

6. 系统检测到状态变更
7. 自动创建 Feature 对象
8. 触发需求分析师 AI 团队

预期结果: 产品负责人快速决策，决策后自动流转到下一阶段
```

### 5.4 场景四：需求分析 + 方案输出

```
角色: 需求分析师 AI 团队 + 产品设计师
触发: Feature 进入需求分析状态

流程:
1. AI 团队分析特性需求：
   - 细化用户故事
   - 制定 5 条验收标准（Eval）
   - 提出 5 套实现方案（含优缺点、成本、工期、风险）

7. 在飞书表格中更新状态为"通过需求评审"

预期结果: AI 快速输出标准化需求文档，真人高效评审确认

9. 测试 AI 读取需求 + 技术设计

11. 测试通过后发送验收邮件给产品设计师
12. 产品设计师验收，状态变更为"已完成"

```

## 8. 数据字典

### 8.1 状态枚举

| 状态域 | 枚举值 | 说明 |
|--------|--------|------|
| 意向状态 | 新意向、评估中、P0-高价值、P1-中价值、P2-低价值、已批准、已淘汰、需要补充信息 | 商机在意向池中的状态 |
| 特性状态 | 特性池中、优先级排序中、已批准、需求分析中、已淘汰 | 特性在看板中的状态 |
| 需求状态 | 草稿、评审中、已批准、技术设计中、开发中、测试中、已完成、已驳回 | 需求的完整生命周期 |
| 技术设计状态 | 草稿、架构评审中、已批准、编码中、已完成 | 技术设计的状态 |
| 测试状态 | 草稿、执行中、测试通过、测试失败、已验收 | 测试的状态 |
| **Agent 状态** | pending_register（未注册）、pending_claim（待认领）、active（活跃）、suspended（暂停） | Agent 入驻生命周期 |
| **规范审批状态** | pending（待审批）、approved（已批准）、rejected（已拒绝） | 阶段规范的审批流程 |

### 8.2 优先级枚举

| 值 | 标签 | 颜色 | 说明 |
|----|------|------|------|
| P0 | 高价值 | 🔴 红色 | 必须做，影响核心业务 |
| P1 | 中价值 | 🟡 黄色 | 应该做，有价值但非紧急 |
| P2 | 低价值 | 🟢 绿色 | 可以做，资源允许时 |
| - | 已淘汰 | ⚫ 灰色 | 不做 |
